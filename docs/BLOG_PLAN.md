# AGSP ブログ連載計画

「AIにコードを書かせる」から「AIでシステムを構築する」へ。
AGSP (AI General Staff Protocol) の思想と実践を解説する全12回の連載構成案。

## 第1部：思想と準備 (The Doctrine)

### 第1回：なぜ私たちは「Vibe Coding」に敗北するのか？
- コード生成AIの罠。「動くけど直せない」地獄の正体。
- "Don't fix the code. Update the Spec." 宣言。
- AI開発における「技術的負債」の新しい形。

### 第2回：AGSP（AI参謀本部プロトコル）の全貌
- 大本営・参謀本部・前線部隊の役割分担。
- GitHub + Google Drive + NotebookLM を連結するアーキテクチャ。
- 人間がやるべきは「承認」のみである。

### 第3回：【参謀本部の設営】NotebookLMを「第二の脳」にする
- 何を読み込ませるべきか（仕様書、ログ、憲法）。
- 「Audio Overview」をデバッグ会議に使う方法。
- RAGとしてのNotebookLMの真価。

### 第4回：【司令部の構築】Gemini Gems / Claude Projects の設定
- AIに「人格」ではなく「階級と軍規」を与えるSystem Instruction。
- コピペで使える「参謀長プロンプト」の解説。

## 第2部：実践・HAKONIWA戦記 (The Operation)

### 第5回：作戦開始 - 最初の仕様書（Spec）の書き方
- AIが読みやすいMarkdown仕様書とは？
- 曖昧さを排除する「要件定義」のプロンプト技術。
- 「憲法（Constitution）」の重要性。

### 第6回：実装 - Claude Code / Cursor への「正しい命令」
- 「作って」ではなく「仕様書Ver1.2に従え」と言う。
- MCP（Model Context Protocol）を使った「書き込み制限」の導入。

### 第7回：【最重要】「逆流同期（Reverse Sync）」の実演
- エラー発生時の対処フロー完全マニュアル。
- VS Codeでバックスペースキーを押すのを我慢し、Driveを修正する実例。
- ログを参謀本部に持ち帰る儀式。

### 第8回：HAKONIWA開発日誌 - キャラクター一貫性の保持
- 「キャラが設定を忘れる」問題を、参謀本部（NotebookLM）の監査で防いだ事例。
- 長期記憶とコンテキストウィンドウの戦い。

## 第3部：高度な戦術 (Advanced Tactics)

### 第9回：ローカルLLM部隊（Llama/Gemma）の運用
- APIコストを抑えるための「推論のオフロード」。
- 24GB VRAMを使い倒すための役割分担。

### 第10回：MCPによる「強制力」の実装
- 精神論ではなく、システムとして「仕様書を見ないとcommitできない」仕組みを作る。
- 自作MCPサーバーの可能性。

### 第11回：ドキュメントの「CI/CD」
- 仕様書が更新されたら、自動的にテストコードが生成されるパイプライン。
- ドキュメント駆動開発（DdD）の最終形態。

### 最終回：総監督（あなた）がこれからすべきこと
- コーダーからディレクターへの転身。
- AI時代のエンジニアの価値とは「真実（仕様）」の定義にある。
